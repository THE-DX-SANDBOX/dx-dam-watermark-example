# Security Policy

## Supported Versions

Security fixes are best-effort and are expected to target the current default branch.

| Version | Supported |
| --- | --- |
| `main` | Yes |
| older branches and tags | No |

## Reporting a Vulnerability

Do not open public GitHub issues for suspected security vulnerabilities.

Instead:

1. Contact the repository maintainers privately through GitHub.
2. Include a clear description of the issue, affected files or flows, reproduction steps, and impact.
3. If possible, include a proposed mitigation or patch direction.

When a report is received, maintainers will triage it, validate impact, and determine the appropriate remediation and disclosure path.

## Scope

Security reports are especially helpful for:

- credential handling
- secret management
- unsafe defaults in deployment manifests or scripts
- authentication and authorization problems
- exposed administrative or internal service endpoints
- dependency or supply-chain risks with clear impact

## Responsible Disclosure

Please avoid publishing exploit details before maintainers have had a reasonable chance to investigate and address the issue.
